Blue-Green Deployment Guide

Blue-Green deployment is a technique that reduces downtime and risk by running two identical production environments, only one of which (e.g. Blue) is active at any time.

1 Start the Infrastructure

Apply the Redis service and colordeploy namespace first, followed by the original **Blue** version:

kubectl apply -f k8s/common/namespac.yaml
kubectl apply -f k8s/common/redis-deployment.yaml
kubectl apply -f k8s/common/redis-service.yaml
kubectl apply -f k8s/blue-green/deployment-blue.yaml
kubectl apply -f k8s/blue-green/service.yaml
2 Spin Up Green Environment

Deploy the new, upgraded **Green** environment. At this point, the service is still targeting the Blue pods, allowing you to test Green in isolation:

kubectl apply -f k8s/blue-green/deployment-green.yaml
3 Cutover Traffic

Start the **Traffic Tester** in the previous tab. Then, execute the switch by changing the service selector. Watch as 100% of your dashboard traffic instantly shifts to **Green** without dropping a single request!

kubectl patch service colordeploy-active-service -n colordeploy -p '{"spec":{"selector":{"color":"green"}}}'
4 Rollback (if necessary)

If something goes wrong, you can instantly rollback to the Blue version by pointing the selector back:

kubectl patch service colordeploy-active-service -n colordeploy -p '{"spec":{"selector":{"color":"blue"}}}'

Manifest Configuration

k8s/blue-green/service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: colordeploy-active-service
  namespace: colordeploy
spec:
  ports:
    - port: 80
      targetPort: 3000
  selector:
    app: colordeploy
    color: blue # <--- Swap this to "green" to switch environments!

Canary Deployment Guide

A canary deployment consists of routing a small fraction of traffic to a new version (canary) to verify its performance and stability before rolling it out to 100% of users.

1 Establish the Stable Base

Deploy the stable production pods (9 replicas) and the shared Canary service:

kubectl apply -f k8s/common/namespac.yaml
kubectl apply -f k8s/common/redis-deployment.yaml
kubectl apply -f k8s/common/redis-service.yaml
kubectl apply -f k8s/canary/deployment-stable.yaml
kubectl apply -f k8s/canary/service.yaml
2 Launch Canary Pod

Introduce the Canary version (1 replica). Because it has the matching label app: colordeploy-canary, the service routes ~10% of traffic to it:

kubectl apply -f k8s/canary/deployment-canary.yaml
3 Verify Telemetry

Launch the **Traffic Tester**. Observe that roughly 10% of requests go to **Canary** (shown in yellow). Check the Latency and Error charts to verify the canary version is healthy!

4 Final Promotion

If telemetry is clean, promote the new version completely by updating your stable deployment image and scaling down the canary pod:

kubectl set image deployment/colordeploy-stable app=colordeploy-app:v2.0.0 -n colordeploy
kubectl scale deployment/colordeploy-canary --replicas=0 -n colordeploy

Manifest Configuration

k8s/canary/service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: colordeploy-canary-service
  namespace: colordeploy
spec:
  ports:
    - port: 80
      targetPort: 3000
  selector:
    app: colordeploy-canary # <--- Selects BOTH stable and canary deployments!
